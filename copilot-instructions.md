# Copilot Instructions for STM32 Template (STM32L1)

## Project context
- This is a bare-metal / HAL-based STM32L151 embedded firmware project.
- Primary source areas:
	- `Application/src/` and `Application/inc/` for shared app modules.
	- `Application/App/` for primary application logic.
	- `Core/Src/` and `Core/Inc/` for Cube-generated HAL init and IRQ handlers.
	- `Application/ClassB/` for safety/self-test code.
- Build artifacts are in `Debug/`; do not treat generated files as source of truth.

## Coding style expectations
- Use C99-compatible C.
- Keep existing naming conventions and file/module boundaries, especially as they are shown in template.c/h.
- Use `Notes/coding_style.md` as the canonical style reference when creating or refactoring C modules.
- Prefer small, single-purpose functions over large monolithic logic.
- Prohibit dynamic allocation (`malloc/free`) unless explicitly required and side-effects are well understood.
- Minimize stack usage in interrupt context.
- Use `static` for file-local helpers and variables, and `const` where possible.
- Avoid floating-point variables and math in time-critical paths unless required and a MCU FPU is available.

## Safety and embedded constraints
- Assume code may run in ISRs and cooperative task loops.
- Do not block in interrupt handlers.
- Preserve watchdog-friendly behavior (no long unbounded loops).
- Preserve timing-sensitive logic (zero-cross detection, heater/pump timing, ADC sampling cadence).
- Keep register/HAL interactions deterministic and explicit.
- Do not silently change ClassB-related behavior.

## Where to put changes
- New application behavior should usually go under `Application/App/src/` and headers in `Application/App/inc/`.
- Generic app utilities belong in `Application/src/` and `Application/inc/`.
- Only modify `Core/Src/*.c` if hardware init/interrupt behavior must change.
- Avoid editing autogenerated sections in Cube-generated files unless requested; keep USER CODE sections intact.

## Testing and verification guidance
- When changing control logic, include a brief test plan in the response:
	- expected normal behavior,
	- edge/fault behavior,
	- timing impact considerations.
- Prefer compile-time-safe changes with clear error handling.
- If adding macros/constants, place them in the closest module header and avoid magic numbers.

## Documentation expectations
- Add concise comments only where logic is non-obvious (state transitions, timing math, safety checks).
- Update relevant notes in `Notes/` when behavior or timing assumptions change.
- In responses, reference exact files touched and summarize risks.

## Copilot response behavior
- For code reviews, prioritize:
	1. safety regressions,
	2. timing/ISR risks,
	3. state-machine correctness,
	4. hardware interaction side effects,
	5. test gaps.
- Prefer minimal diffs that preserve existing behavior unless change is explicitly requested.
- Ask before making cross-cutting architectural refactors.
